## Codebase Confirmations

- **Shield field**: `userVillageProfile.shieldActiveUntil` ŌĆö type `text` (nullable ISO datetime string). "Has shield" is determined by: `profile.shieldActiveUntil ? new Date(profile.shieldActiveUntil) > new Date() : false`. Shield is NOT consumed on attack ŌĆö it blocks the attack and remains active until expiry. Shield is granted for 24h via `/village/shield` (costs stars) or via plinko milestone (configurable hours via `shield_reward_hours` village_config key).

- **Village buildings / stars**: `userVillageProgress` table has fields `building1Stars` (col `b1_stars`) through `building5Stars` (col `b5_stars`), all `integer default(0) notNull`. Each goes 0ŌåÆ4 (max 4 stars per building, 20 total to complete a village). There is NO explicit minimum constraint preventing stars from going below 0 at the DB level ŌĆö the `.default(0)` just sets initial value. The current attack system drops victim's LEVEL (not stars), so building stars have never been decremented. For the new system where attacks steal stars from buildings, we need to ensure stars never go below 0 in application logic.

- **Lucky Drop attack cards**: 
  - State declared: line 246 ŌĆö `const [attackCards, setAttackCards] = useState(0)`
  - Incremented on reward: line 829 ŌĆö `setAttackCards((c) => Math.min(c + 1, 3))`
  - Displayed: lines 964-968 ŌĆö 3 sword icon slots (`/images/sword.png`), opacity 1 if `i < attackCards`, else 0.2
  - **NO existing code triggers when attackCards reaches 3.** The state caps at 3 but nothing fires an attack flow. This is where we need to add the trigger (navigate to attack page or show attack modal).
  - Animation queue: `queueAnim` (line 261) pushes "shield"|"card" to `animQueueRef`, then calls `playNextAnim` if nothing is playing. `playNextAnim` (line 250) shifts from queue, sets corresponding show state. Animations play sequentially, one at a time.

- **Coin balance**: The `transactions` table stores coin packs. Key fields: `coinsReceived`, `coinsRemaining`, `status` ("active"|"bonus_round"|"completed"). Coins are spent by finding an active transaction with sufficient `coinsRemaining` and decrementing it. Helpers in `shansi-backend/src/socket/index.ts`:
  - `getCoinBalance(userId)` ŌĆö sums `coinsRemaining` across all active/bonus_round transactions
  - `deductCoins(userId, amount)` ŌĆö drains from oldest active tx first, returns boolean
  - `addCoins(userId, amount)` ŌĆö adds to first active tx or creates new one
  These are currently only used by air hockey (socket). Village upgrade uses raw SQL: `UPDATE transactions SET coins_remaining = coins_remaining - ${cost} WHERE id = ${tx.id}`.

- **Existing attack endpoint**: `POST /village/attack` (village.ts lines 181-253) is FULLY IMPLEMENTED with:
  - Requires `victimId` + `cardIds[]` (array of userCard IDs)
  - Validates `cardIds.length === attack_cards_needed` (config, default 3)
  - Checks attacker & victim same level
  - Shield check on victim (blocks if shielded)
  - Verifies attacker owns the cards (checks `userCards` table by ID + userId)
  - Rolls success/fail based on `attack_success_rate` config (default 50%)
  - ALWAYS consumes the cards (deletes from `userCards`)
  - On success: awards `attack_star_bonus` stars (default 15) to attacker, drops victim's level by 1
  - Logs to `villageAttacks` table
  - **Problem for new system**: This endpoint uses the LEGACY card system (villageCards/userCards). For the new system where Lucky Drop grants attack cards (not legacy shop cards), we need a DIFFERENT mechanism. Options: (A) extend this endpoint to accept a "use attack charges" mode without cardIds, or (B) create a new endpoint. Recommendation: Create a new `POST /village/attack-v2` endpoint that deducts attack charges from a counter rather than consuming card rows.

- **User cards table**: `userCards` has `id`, `userId`, `cardId` (FK to villageCards). Each row = one card instance. To consume 3, you delete 3 rows by their `id`. The existing attack endpoint already does this: `await db.delete(userCards).where(eq(userCards.id, cid))` per card. However, for the NEW system, attack cards are tracked as a simple counter in Lucky Drop state (frontend only). We need a backend-persisted attack card count ŌĆö either a new column on `userVillageProfile` (e.g. `attack_charges integer default 0`) or continue using `userCards` rows.

## Risks & Open Questions

1. **Attack card persistence**: Currently `attackCards` in Lucky Drop is ephemeral (useState). Need to decide: store on backend profile (recommended) vs localStorage (fragile). Task breakdown assumes backend.
2. **Which building to attack?** Should attacker choose, or is it random? Current plan: attacker taps a building on EnemyVillageView. Alternative: random selection with animation.
3. **What if all victim buildings are at 0 stars?** Attack should still succeed but award 0 stars (or prevent targeting that victim entirely in the targets list).
4. **Race condition**: Two users attack same victim's same building simultaneously. Need atomic decrement with floor at 0: `UPDATE ... SET b{N}_stars = MAX(0, b{N}_stars - 1) WHERE ...`
5. **Level mismatch after attack**: If victim's stars drop, do they lose their village progress? Current attack endpoint drops level ŌĆö new system should only drop building stars, NOT level. Levels are tied to village completion, not individual stars.
6. **Notification system**: No real-time push notification infra exists. Victim only sees attack result on next village page load. Acceptable for MVP?
7. **Legacy vs new attack endpoint**: Should we deprecate `POST /village/attack` or keep both? Recommend keeping legacy but not exposing it in new UI.
8. **Sound files needed**: May need 1-2 new audio files (attack-launch, explosion-hit). Can we reuse `battle.mp3`?
